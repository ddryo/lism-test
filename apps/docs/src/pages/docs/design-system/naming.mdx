# Naming Rules

- 気をつけていること
	- BEM系と、WordPressコア機能で使用されるクラスとの衝突がない命名規則を採用。


## クラス名の命名規則

### モジュールの分類と命名規則

|分類|形式|例|
|---|---|---|
|レイアウトモジュール<br /><small>主要なレイアウト要素。</small>|`.l--{moduleName}`|`.l--box`, `.l--layer`|
|エレメントモジュール<br /><small>レイアウトにおける最小単位。</small>|`.e--{moduleName}`|`.e--avatar`,`.e--icon`|
|コンポーネントモジュール<br /><small>`l--`, `e--`で組めるようなもの。</small>|`.c--{moduleName}`|`.c--banner`, `.c--button`|
|ブロックモジュール<br /><small>`l--`, `c--`で組めるようなもの。</small>|`.b--{moduleName}`|`.b--navMenu`, `.b--section`|

{/* |インライン書式 ? |`.i--{書式名}` or `.f--{書式名}`|`.i--marker` or `.f--marker`| */}

Memo: tab,accordionを `d--`(disclosure) とする？

補足

- `{moduleName}`の部分はキャメルケース.
    - 例: `.l--ratioGrid`
- Modifierは`.{prfx}--{moduleName}` & `.{prfx}--{moduleName}--{modifierName}`。
	- 例: `.l--layer.l--layer--media`
	- ※ 要検討。
		- `.l--layer--media`だけでいいか。
		- `.layer:media` とかユーティリティのような形式でもいいかも。
		- もしくは、データ属性。`.l--layer[data-layer="media"]`
		- `_media`, `-media`  ← こういうのは `l--` に対してなのか `c--` に対してなのかわからない
- 各モジュールの子要素はBEM同様2つの`__`で連結する。
	- `.{prfx}--{moduleName}__{elemName}`
	- 例: `.b--section__body`



#### 各モジュールの分類と関係性について

- `.l--`系: レイアウトプリミティブな要素。
	- 他の`.l--`と併用はNG。(`.l--`の子要素 に `.l--`が入ってくるのはOK.
- `.e--`系: それ単体では意味として薄く、デザインの最小単位となるような要素。
- `.c--`系: `.l--`, `.e--`を組み合わせて作れるような要素。デザインとしてより具体的な意味をもつモジュールです。
- `.b--`系: `.l--`,`.c--`を組み合わせて作れるような要素。デザインとしてより具体的な意味をもち、かつ`.c--`よりも構造が複雑なモジュールです。

memo: `.c--badge` は 見た目としてはとても小さいが、内部にElemenであるアイコンがくることもあり、また用途としても明確なので `.c--` としている。


### Utilityクラスの分類と命名規則

|分類|形式|例|
|---|---|---|
|状態管理クラス|`.is--{stateName}`|`.is--flow`, `.is--constrained`, `.has--gutter`|
| |`.has--{stateName}`|`.is--flow`, `.is--constrained`, `.has--gutter`|
|プロパティ管理クラス<br /><small>特定のプロパティに対する値を指定するためのもの。</small>|`.-{prop}:{value}`|`.-ta:c`, `.-fz:xl`, `.-p:50`|
| |`.-{プロパティ}:`|`.-w:` (→ `--w`変数を受け取る)|

	- ネガティブ(マイナス)値の表記もそのままで。
		- 例: `.-prop:-1`

#### data属性について

「特定のモジュールに関する機能的なバリエーション」にあたるものはクラスではなくdata属性で管理する。

- 例: `.l--frame[data-ratio="4/3"]`, `.e--badge[data-variant="xxx"]`など


### その他、Lism管理外での推奨規則

Lismで管理する範囲外だけどこんな感じで命名すると統一感でるよ、というもの。

|分類|形式|例|
|---|---|---|
|エリア|`.a--{areaName}`|`.a--header`,<br />`.a--sidebar`,<br />`.a--article`|
|ページ種別<br /><small>WordPressではbodyに吐き出されるクラスが標準で用意されてるので、そっちを使う。</small>|`.p--{type}--{id\|slug}`|`.p--front`, `.p--archive`, `.p--post--{id}`, `.p--page--{slug}`|

## 単語の省略について

ユーティリティクラス名、CSS変数名、コンポーネントに渡すプロパティ名に関しての省略について。

- CSSプロパティ名に関しては、基本的には[Emmetの省略形](https://docs.emmet.io/cheat-sheet/) に揃える。
	- background → `bg`
	- background-color → `bgc`
	- border → `bd`
	- `text-align:center;` のユーティリティクラス → `.-ta:c`
- ※ Emmet に存在しないものどうする問題あり。
	- grid-template-area → gtc
	- row-gap, column-gap → rowg, clmng
	- 論理値プロパティ系どうする?
		- padding-inline-start とか
		- margin-block とか


その他、わかりやすい単語、よくあるものは省略？
- text → txt
- button → btn


column, columnsの省略について

- colm, colms → あり。ただし Emmetさんが colm = columns としている...。
- clmn, clmns → col,colsの次にけっこう一般的？
- clm, clms → これもありか。 でも他の単語とも被りやすそう？
- そもそもどこで使う
